## 1. この資料の目的

本資料は、アジャイル開発を体系的に学ぶための **疑似プロジェクト教材** として作成しています。

アジャイル学習では、実際の現場に近い題材のほうが理解が進みますが、
本物のプロジェクト資料は機密情報が多く扱いにくいため、教育目的として **架空の図書館システムのデジタル化** を題材としています。

この疑似プロジェクトは以下を目的としています：

* アジャイルの概念（Epic、Story、Backlog、Increment）の理解
* スプリント計画・見積り・優先度設定などの実務感覚の習得
* AS-IS / TO-BE、ユースケース、業務フローなど上流工程の分解練習
* 変更要求を扱う演習（アジャイル特有のバックログ更新経験）
* Docs-as-Code によるドキュメント体系の学習

---

## 2. 題材として「図書館」を採用した理由

図書館は IT 初学者にも理解しやすく、かつシステム化ポイントが多いドメインのため、学習教材として最適です。

### 📌 採用理由

* **Excel運用 → システム化** の変換が明確で、改善点が見えやすい
* 業務がシンプル（貸出・返却・検索・予約）で説明しやすい
* 拡張（通知、分析、アプリ化）も容易で、アジャイル演習に向く
* 学習者ごとに多様な要求追加が可能
* 技術要素（API、DB、UI/UX）も幅広く扱える

教育現場・研修・オンボーディングで長年使われている題材でもあります。

## 3. 疑似プロジェクトで扱う主な資料

この疑似プロジェクトでは、アジャイル開発の一連の流れを学べるよう、以下の資料群を整備しています。

### ● 00_docs（現状分析）

* 図書館のExcel運用の業務プロセス
* Pain point（課題）の整理

### ● 01_vision（戦略・イニシアチブ）

* デジタル化の目的と期待効果
* イニシアチブ（Initiative）の定義
* アーキテクチャ概要

### ● 02_backlog（要求の分解）

* Epic（価値のまとまり）
* User Story（ユーザー価値）
* Task（実装作業）
* MoSCoW / 優先順位
* UI プロトタイプ
* ドメインモデル

### ● 03_specs（設計資料）

* 詳細設計
* 実装
* 単体テスト

### ● 04_release（リリース計画）

* MVP 定義
* インクリメント例
* スプリント計画

これらの資料を通して、**要件探索 → 分解 → 優先度付け → 実装 → ふりかえり** の流れを疑似体験できます。

## 4. この疑似プロジェクトを使って学べること

### ✔ アジャイルの基本構造

* Epic / Story / Task の粒度
* Backlog の積み上げと更新
* インクリメンタル開発の進め方

### ✔ ユーザー中心の要求分析

* AS-IS / TO-BE の比較
* Pain から価値を抽出する方法
* ユースケースとストーリーの違い

### ✔ スプリント実務

* 見積り（ストーリーポイント）
* 優先度付け（MoSCoW、WSJFなど）
* レトロスペクティブ

### ✔ チームコラボレーション

* プロダクトオーナー／開発者／スクラムマスターの役割
* 合意形成と仕様調整の進め方

### ✔ Docs-as-Code の活用

* ディレクトリ構成による文書整理
* Markdown ベースの継続的な更新

## 5. この資料の使い方（学習プラン例）

### 🔰 初学者向け（1〜2週間）

1. AS-IS の理解
2. Epic と Story の読み込み
3. ストーリーの優先度付け練習
4. スプリント 1 の計画を作る

### 🧑‍💻 実務者向け（1ヶ月）

1. 自分で追加要求を考えてバックログ化
2. Story の受入条件（AC）を記述
3. ドメインモデルと画面遷移を設計
4. ふりかえり（KPT）を記述

### 👥 チーム研修向け（1〜3ヶ月）

* ロールプレイ（PO/SM/Dev）
* ストーリー分割ワーク
* UI プロトタイピング
* インクリメント発表会

アジャイルの全体像を体験的に理解できる構成になっています。

## 6. なぜ疑似プロジェクト形式が有効なのか

### 📌 1. 現場のように「曖昧さ」や「変更」を扱える

アジャイルが得意とするのは変化への適応であり、疑似プロジェクトでも後で変更要求を入れることで学習可能です。

### 📌 2. 実務と同じように資料を使い続けられる

実務ではドキュメントは静的ではなく、常にアップデートされます。
本教材も同様に、参加者が自ら追加・修正していく前提の構成にしています。

### 📌 3. 安全な環境で失敗できる

本物のプロジェクトと異なり、失敗しても誰にも影響しません。
スプリント計画のミス、過剰な分割、見積もりのズレなどを体験的に学べます。
