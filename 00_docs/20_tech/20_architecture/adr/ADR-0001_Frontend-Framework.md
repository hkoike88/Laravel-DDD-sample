# ADR-0001: フロントエンドフレームワーク（React）

## ステータス

採用

## コンテキスト

図書館デジタル化システムのフロントエンド開発において、以下の要件を満たすフレームワークが必要である。

- SPA（Single Page Application）として動的な UI を構築
- TypeScript との親和性が高い
- 豊富なエコシステム（ライブラリ・ツール）
- 長期的なメンテナンス性
- チームの学習コスト

## 決定

**React 18.3** を採用する。

### 採用理由

1. **エコシステムの充実**
   - 状態管理、ルーティング、フォーム処理など、成熟したライブラリが豊富
   - TanStack Query、React Router、React Hook Form など高品質なツールが利用可能

2. **TypeScript サポート**
   - 公式で TypeScript を完全サポート
   - 型定義が充実しており、型安全な開発が可能

3. **学習リソースの豊富さ**
   - 日本語ドキュメント・書籍・記事が豊富
   - 学習プロジェクトとして適切

4. **柔軟性**
   - 「ライブラリ」であり、アーキテクチャを自由に設計可能
   - 必要な機能だけを選択して導入できる

5. **長期サポート**
   - Meta（旧 Facebook）が継続的に開発
   - 大規模な採用実績があり、今後も安定したサポートが期待できる

## 比較検討

| 項目 | React | Vue.js | Angular | Svelte |
|------|-------|--------|---------|--------|
| 学習曲線 | 中 | 低 | 高 | 低 |
| TypeScript | ◎ | ○ | ◎ | ○ |
| エコシステム | ◎ | ○ | ◎ | △ |
| パフォーマンス | ○ | ○ | ○ | ◎ |
| 求人・情報量 | ◎ | ○ | ○ | △ |
| 柔軟性 | ◎ | ○ | △ | ○ |

### 不採用理由

- **Vue.js**: エコシステムが React ほど成熟していない。Options API と Composition API の混在
- **Angular**: 学習コストが高い。フルスタックフレームワークで柔軟性に欠ける
- **Svelte**: エコシステムが発展途上。大規模採用実績が少ない

## 結果

### メリット

- 豊富なライブラリ選択肢により開発効率向上
- 関数コンポーネント + Hooks による宣言的な記述
- 将来的な人材確保が容易

### デメリット

- フレームワークではなくライブラリのため、構成を自分で決める必要がある
- 状態管理など、追加ライブラリの選定が必要

### リスクと対策

| リスク | 対策 |
|--------|------|
| アーキテクチャの統一性 | Feature-based アーキテクチャを標準として定義 |
| ライブラリ選定の迷い | 本 ADR シリーズで推奨ライブラリを明確化 |
